## 1. VISIÓN GENERAL DEL PROYECTO

**Nombre**: EUREKA BANK - Cliente SOAP/REST
**Stack**: React 18 + Vite + TypeScript + CSS
**Objetivo**: Cliente web para operaciones bancarias que soporta 4 backends diferentes (SOAP Java, REST Java, SOAP .NET, REST .NET)

### Arquitectura Soportada
```
Frontend (React 5173) 
    ↓ Proxy Vite (dev)
    ↓ Direct URLs (prod)
    ↓
4 Backends Remotos:
  - SOAP Java (8091): http://209.145.48.25:8091/ROOT/CoreBancarioWS
  - REST Java (8090): http://209.145.48.25:8090/resources/corebancario
  - SOAP .NET (8092): http://209.145.48.25:8092/CoreBancarioWS
  - REST .NET (8093): http://209.145.48.25:8093/resources/corebancario
```

## 2. PROGRESO COMPLETADO

### ✅ Problemas Resueltos

#### A. Invalid Hook Call Error
**Problema**: Servicios llamaban `useBackend()` dentro de funciones no-componente
**Solución**: Refactorizar ALL servicios para recibir `backendActual` como parámetro
**Archivos Corregidos**: 
- `apiService.ts` - Centraliza routing a SOAP/REST
- `soapService.ts` - Todos los métodos reciben `backendActual`
- `restService.ts` - Todos los métodos reciben `urlBase`

**Lección Aprendida**: React hooks SOLO funcionan en componentes o custom hooks, NUNCA en servicios.

---

#### B. CORS Policy Blocks
**Problema**: Browser bloqueaba requests: "Access to fetch has been blocked by CORS policy"
**Root Cause**: Backends en IP externa (209.145.48.25), frontend en localhost:5173, sin CORS headers

**Solución Implementada**:
1. **Vite Proxy** (`vite.config.js`):
   ```javascript
   proxy: {
     '/api/soap-java': { target: 'http://209.145.48.25:8091', rewrite: path => path.replace(/^\/api\/soap-java/, '') },
     '/api/rest-java': { target: 'http://209.145.48.25:8090', rewrite: path => path.replace(/^\/api\/rest-java/, '') },
     '/api/soap-dotnet': { target: 'http://209.145.48.25:8092', rewrite: path => path.replace(/^\/api\/soap-dotnet/, '') },
     '/api/rest-dotnet': { target: 'http://209.145.48.25:8093', rewrite: path => path.replace(/^\/api\/rest-dotnet/, '') }
   }
   ```

2. **Environment Detection** (`backendConfig.ts`):
   ```typescript
   const isProduction = import.meta.env.PROD;
   // Dev: usar proxy /api/xxx (Vite redirige)
   // Prod: usar direct URLs (9.145.48.25:8091)
   ```

**Archivos Modificados**: 
- `vite.config.js` - 4 proxy entries
- `backendConfig.ts` - isProduction branch

**Validación**: Requests en dev usan `http://localhost:5173/api/soap-java/...` → Vite convierte a `http://209.145.48.25:8091/...`

---

#### C. SOAP XML Response Parsing
**Problema Original**: Servidor devolvía `<return>1</return>` pero validación buscaba solo strings específicos

**Problemas Encontrados y Solucionados**:

1. **Login Response**: `<return>Exitoso</return>`
   - Solución: Agregar validación para "Exitoso" y "exitoso"

2. **Deposito/Retiro/Transferencia Response**: `<return><estado>1</estado><saldo>6950.0</saldo></return>`
   - Solución: Agregar validación para `<estado>1</estado>`

3. **Movimientos Response**: Múltiples `<return>` tags (cada uno es un movimiento)
   - Problema Inicial: Buscaba tags `<movimiento>` que no existen
   - Solución: Iterar sobre TODOS los `<return>` tags y extraer campos de cada uno
   ```typescript
   const returnNodes = xmlDoc.getElementsByTagName('return');
   for (let i = 0; i < returnNodes.length; i++) {
     const nodo = returnNodes[i];
     if (cuenta && nromov) {  // Validar que sea movimiento
       movimientos.push({...});
     }
   }
   ```

**Archivos Modificados**: `soapService.ts` - 4 métodos validación mejorada

**Lección**: SOAP devuelve múltiples formatos según la operación. Necesita logging exhaustivo para discovery.

---

#### D. REST Java Error Handling
**Problema**: Servidor devolvía HTTP 500 pero con JSON válido `{"estado":-1,"saldo":-1.0}`
**Root Cause**: Código original: `if (!respuesta.ok) throw Error` → nunca llegaba a parsear JSON

**Solución**:
```typescript
try {
  const jsonParse = JSON.parse(contenido);
  return jsonParse;  // Devolver JSON INCLUSO si status 500
} catch (e) {
  if (!respuesta.ok) throw new Error(...);  // Solo thrower si no es JSON
}
```

**Archivo Modificado**: `restService.ts` - `realizarSolicitud()`

**Lección**: No asumir que status 500 = sin datos. Algunos servidores devuelven errores en JSON con status 500.

## 3. PROBLEMAS CONOCIDOS Y SOLUCIONES

### 🔴 REST Java Devuelve estado:-1
**Síntoma**: `POST /deposito` responde con `{"estado":-1,"saldo":-1.0}`
**Análisis**:
- ✅ Login funciona (`resultado: Exitoso`)
- ❌ Deposito falla (estado: -1)
- Posibles causas:
  1. Token/sesión no se mantiene (pero rest.ts guarda token en header Authorization)
  2. Backend REST Java requiere parámetros adicionales
  3. Datos mal formateados

**Estrategia de Debug**:
1. Verificar en Postman que REST Java deposito funciona
2. Comparar request exacto en Postman vs desde app
3. Si falla en ambos: problema del servidor, no de la app
4. Si funciona en Postman: problema de formato de datos o headers

**Solución Temporal**: SOAP Java funciona, seguir usando SOAP hasta resolver REST Java.

---

### ⚠️ REST Java requiere Token?
**Hallazgo**: Login devuelve `resultado: Exitoso` pero NO devuelve token
**Código Actual**:
```typescript
if (respuesta?.token) {
  tokenAutenticacion = respuesta.token;
}
```
**Problema**: Si REST Java no devuelve token, `tokenAutenticacion` permanece vacío
**Solución Potencial**:
1. Usar session cookies (servidor puede mantener sesión)
2. O el servidor requiere Bearer token que no se devuelve en login

---

### ⚠️ Validación de Movimientos Incompleta
**Síntoma**: Algunos `<return>` podrían no ser movimientos
**Actual**: Filtramos solo si `cuenta && nromov` existen
**Mejora Futura**: Validar campo por campo y usar valores por defecto

## 6. DATA FLOW EXAMPLES

### Login Flow
```
LoginPage.jsx 
  → handleLogin(usuario, password)
  → apiService.autenticar(usuario, password, backendActual)
    → if (backendActual.protocolo === 'SOAP')
        → soapService.autenticar()
          → realizarSolicitudSoap()
          → Valida <return>Exitoso</return>
      else
        → restService.autenticar()
          → realizarSolicitud()
          → Guarda token si existe
          → Valida resultado: 'Exitoso'
  → navigate('/principal') si exito
```

### Deposito Flow
```
DepositoView.jsx
  → handleProcesar(cuenta, importe)
  → Valida campos
  → apiService.registrarDeposito(cuenta, importe, backendActual)
    → Convierte importe string → number con parseFloat()
    → if SOAP: soapService.registrarDeposito()
    → if REST: restService.registrarDeposito()
  → Muestra resultado
```

### Movimientos Flow
```
ConsultaView.jsx
  → handleConsultar(cuenta)
  → apiService.obtenerMovimientos(cuenta, backendActual)
    → if SOAP:
        → Obtiene XML
        → Itera <return> tags
        → Extrae campos de cada uno
        → Retorna array Movimiento[]
    → if REST:
        → Obtiene JSON
        → Busca en response.movimientos || response.data || response.items
        → Retorna array Movimiento[]
  → Renderiza tabla
```

## 7. PROBLEMAS COMUNES Y SOLUCIONES RÁPIDAS

### Problema 1: "Invalid hook call outside component body"
**Síntoma**: Error en console cuando llamas servicio
**Causa**: Servicio intenta usar `useBackend()` o `useContext()`
**Solución**: Pasar `backendActual` como parámetro en lugar de usar hook
```typescript
// ❌ MAL
export const miServicio = () => {
  const { backendActual } = useBackend();  // Error!
}

// ✅ BIEN
export const miServicio = (backendActual) => {
  // usa backendActual
}
```

---

### Problema 2: "CORS policy blocks request"
**Síntoma**: Request en red tab muestra status 0, console error CORS
**Causa**: Proxy no configurado o URL incorrecta
**Solución**:
1. Dev: Verificar `vite.config.js` tiene proxy para ese backend
2. Prod: Verificar `backendConfig.ts` tiene URL directa correcta
3. Verificar en console: `[REST] POST /api/rest-java/...` (dev) vs `http://209.145.48.25:8090/...` (prod)

---

### Problema 3: "respuesta: {estado: -1, saldo: -1.0}"
**Síntoma**: Operación (deposito, retiro) falla
**Diagnóstico**:
1. Verificar que backend selector sea correcto (LoginPage dropdown)
2. Probar en Postman con MISMOS parámetros
3. Si funciona en Postman pero no en app: problema de tipo de dato
   - REST espera `importe: 50.0` (number), no `"50"` (string)
4. Si falla en Postman también: problema del servidor

---

### Problema 4: "No se encontraron movimientos"
**Síntoma**: Tabla vacía en ConsultaView
**Diagnóstico**:
1. Verificar logs en console `[MOVIMIENTOS]`
2. Buscar `[MOVIMIENTOS] Se encontraron 0 movimientos`
3. Mirar `[MOVIMIENTOS] Respuesta XML:` para ver estructura real
4. SOAP: Si ve `<return>` tags pero cuenta mismatch, revisar parsing
5. REST: Si respuesta es `[]` o `{}`, revisar qué devuelve el servidor

**Debug Rápido**:
```typescript
// En soapService.ts obtenerMovimientos():
console.log('[MOVIMIENTOS] returnNodes.length:', returnNodes.length);
console.log('[MOVIMIENTOS] Primer nodo:', returnNodes[0]);
```

## 10. HABILIDADES DEL AGENTE

Cuando trabajes en este proyecto, debes:

### 1. DIAGNOSIS
- [ ] Revisar console logs con prefijos `[SOAP]`, `[REST]`, `[LOGIN]`, etc.
- [ ] Buscar patrón: "Buscando indicadores de éxito" → significa que validación falló
- [ ] Verificar `backendConfig.ts` para URLs correctas según backend
- [ ] Confirmar que componente pasa `backendActual` a servicio

### 2. SOAP DEBUGGING
- [ ] Si "No se encontraron indicadores de éxito": agregar nuevo patrón en validación
- [ ] Si XML parsing falla: revisar estructura con `console.log(respuestaXml)`
- [ ] Si "0 movimientos": iterar sobre TODOS los tags, no asumir nombre

### 3. REST DEBUGGING
- [ ] Si status 500: intentar `JSON.parse(contenido)` antes de thrower
- [ ] Si autenticación falla: verificar si devuelve token y guardar en headers
- [ ] Si campo no existe: usar optional chaining `respuesta?.field?.toLowerCase()`

### 4. CONVERSIONS
- [ ] REST recibe importe como `number`, pasar `parseFloat()`
- [ ] SOAP puede recibir como string, lo convierte internamente
- [ ] Arrays: REST a veces devuelve en `data`, `items`, `movimientos` → intentar todos

### 5. ERROR HANDLING
- [ ] Siempre capturar con try/catch
- [ ] Loguear ANTES y DESPUÉS de operación
- [ ] Devolver objeto `{ exito: boolean, mensaje: string }`
- [ ] NUNCA dejar usuario sin feedback

### 6. ADDING NEW METHODS
1. Decide si aplica solo a algunos backends o todos
2. Agregar a `soapService.ts` O `restService.ts` O ambos
3. Crear validación específica para ese servidor
4. Agregar a `apiService.ts` con routing
5. Loguear con prefijo `[NOMBREOPERACION]`
6. Probar en al menos 2 backends
